## Codebase Patterns
(durable, reusable patterns only — graduate anything stable into skills/ and delete it here)

---
Tactical learnings below, newest last. Hard cap: 100 lines total (enforced by adw/progress.py).
Prune every run: durable patterns -> skills/, harness ambiguity/bugs -> system-repair ticket, stale -> delete.
---

S-017: state.stage already distinguishes "finished" (document/decompose/observe)
from "mid-pipeline interrupted" (one of stage_order) — no new State field needed
to detect a resumable quota interruption; just check `stage in stage_order`.

GH-46: _ensure_work_branch's two paths (existing branch -> checkout, new ->
checkout -b) need different dirty-tree handling — checkout -b silently carries
a dirty tree over, but plain checkout aborts on one. _commit_bookkeeping is a
safe no-op-on-clean-tree helper already used for prd.json bookkeeping; reuse
it before any checkout onto an existing branch instead of inventing a new
stash path (stash risks merge conflicts when the target branch's file has
diverged).

GH-47: state.json is gitignored, so its mtime is a clean heartbeat — git
operations (checkout, commit) never touch it, and save_state rewrites it at
every stage entry. That made it the natural staleness signal for reclaiming a
stranded in_progress ticket, cheaper than threading a lock/process check
through pick_next_story. Split as a pure predicate (reclaim_stale_in_progress,
tickets.py) + thin IO wrapper (reap_stale_in_progress, workflow_runner.py) so
the staleness logic itself needs no tmp-repo fixture to test.

GH-56: any helper that needs the engine's own git remote (self-host check)
must resolve it at MODULE IMPORT time, not inside the function — a test that
monkeypatches `subprocess.run` to fake a different repo's git calls (clone,
fetch) will also stomp the real `git remote get-url origin` call if it's
resolved lazily, since `import subprocess` binds the same module object
everywhere. Also: when faking a paginated API by string-matching a query
param in a test, match on `path.endswith("&page=N")`, not `"page=N" in path`
— `per_page=100` itself contains the substring "page=100", which contains
"page=1" as a prefix, so an `in`-based page=1 check always matches and the
fake "pages" forever.

GH-61: the optional-field pattern (pr_description on StatusBlock) is reusable
for any new stage-handoff data — add to the dataclass, parse leniently
(non-matching shape -> None, never raise), thread it through compose_stage_
prompt by reading the latest iter*_<stage>_output.md with parse_status_block
inside try/except. Kept the new file_manifest section gated to
implement/test/review only (not plan itself) since the manifest only exists
once a plan has run.
