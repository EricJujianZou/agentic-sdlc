# GH-53: Cross-repo self-heal routes harness-level findings to the engine repo

## What shipped

The observer already classifies a failure as `harness`-level vs ticket-level
and posts the diagnosis back to the run's source issue. When the harness runs
cross-repo (a downstream user building their own target repo with their own
fork of the engine), that report only ever reached the *target* repo — the
engine owner never saw it. This adds a best-effort branch that, on a
harness-level finding during a cross-repo run, files a triage-gated
system-repair issue in the engine repo itself, reusing the observer result and
GitHub helpers already in place (no new subsystem, no extra agent call).

### CLI / workflows

`adw/workflow_runner.py`'s `_observe_and_report` now calls
`_file_upstream_repair(story, result)` whenever
`result.classification == "harness" and result.repair`, fired before the
existing `number is None` early-return so it fires even for plain `S-NNN`
tickets with no source issue. `_file_upstream_repair`:

- No-ops immediately if `paths.target_root() == paths.engine_root()`
  (self-hosted run — unchanged behavior).
- Resolves the engine repo via the new `engine_repo_slug()` in
  `adw/github.py`: reads `ADW_ENGINE_REPO` (`owner/repo`) if set (raises
  `GitHubError` if malformed), else falls back to `repo_slug(paths.engine_root())`.
- Computes a 12-hex-char dedup fingerprint (`_repair_fingerprint`) by hashing
  the normalized repair title (falling back to `result.summary`, then
  `story.id`), and skips filing if any open `self-heal-suggested` issue in the
  engine repo already embeds that fingerprint (via a new
  `list_open_issues(label=...)`).
- Files the issue via a new `create_issue(owner, repo, token, title, body,
  labels=...)`, labeled `self-heal-suggested` only — deliberately **not**
  `adw`, since the poller only auto-runs `adw`-labeled issues, so a human must
  triage and label it in before the harness acts on it.
- Wraps the whole thing in `try/except GitHubError`, printing and swallowing
  any failure (missing token scope on the engine repo, network error, etc.)
  so a routing failure never breaks the run.

## Behavior & breaking changes

None — self-hosted runs (`target_root() == engine_root()`) are byte-for-byte
unchanged; this only adds a new branch that's a no-op in that case.

## How it was verified

- "Cross-repo harness finding files an engine-repo issue" →
  `tests/test_observer.py::test_file_upstream_repair_cross_repo_files_issue`,
  `test_observe_and_report_files_upstream_when_cross_repo_harness` — assert
  `create_issue` called with the engine slug and fingerprint marker in the body.
- "Engine slug configurable" →
  `tests/test_github.py::test_engine_repo_slug_from_env`,
  `test_engine_repo_slug_malformed_env_raises`,
  `test_engine_repo_slug_falls_back_to_engine_remote`.
- "Human-label-gated" → same cross-repo test asserts
  `created["labels"] == [SELF_HEAL_LABEL]` and `"adw" not in labels`.
- "Deduped by fingerprint" →
  `test_file_upstream_repair_dedup_skips_existing` — `create_issue` not called
  when `list_open_issues` returns a matching fingerprint.
- "Best-effort, never fatal" →
  `test_file_upstream_repair_swallows_github_error` — `get_token` raises,
  `_file_upstream_repair` returns without raising.
- "Self-hosted unchanged" →
  `test_file_upstream_repair_self_hosted_noop` — `create_issue` not called
  when `target_root() == engine_root()`.
- "Deterministic, no extra agent call" — `_file_upstream_repair` only consumes
  the `result` already returned by the single `run_observer` call in
  `_observe_and_report`; no second `invoke_fn`/observer invocation is
  monkeypatched or exercised in any test.
- Full suite: `uv run pytest -q` → 326 passed.

## Review notes

- The `paths.target_root()/engine_root()` guard sits outside the `try` block
  in `_file_upstream_repair`, so a (very unlikely) exception from those calls
  would propagate rather than be swallowed. Noted by review as non-blocking —
  those calls are reliable and used elsewhere unguarded.
- Dedup fingerprinting hashes the agent-generated repair title, which may
  reword across runs for the same underlying bug; residual duplication is
  accepted since the issue is human-triaged anyway.
