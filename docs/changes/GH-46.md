# GH-46: Sync's uncommitted prd.json write blocks checkout to a pre-existing work branch

## What shipped

A cross-repo poll (`ADW_REPO=design-system python workflows/poll_once.py`)
crashed when the GitHub-sync step rewrote `prd.json` and left it uncommitted
on `main`, then the backlog runner tried to resume a story whose `adw/<id>`
branch already existed from a prior run. `_ensure_work_branch` took the
`git checkout <branch>` path, and git refused to switch over the dirty
`prd.json`, raising `CalledProcessError` and aborting the whole pass. The fix
commits the dirty sync write before switching, only on the existing-branch
path â€” the fresh-branch (`checkout -b`) path is untouched.

## Behavior & breaking changes

`_ensure_work_branch` (`adw/workflow_runner.py:79-85`) now calls
`_commit_bookkeeping("chore: persist sync before resuming work branch")`
immediately before `git checkout <branch>`, but only inside the
`if _git("branch", "--list", branch):` branch (i.e. only when the work branch
already exists). `_commit_bookkeeping` is a no-op on a clean tree, so a resume
with no pending sync write behaves exactly as before. This means a poll can
now create a **local** commit on `main` (or whichever branch sync ran on)
during the syncâ†’resume hand-off â€” it is never pushed or merged, so the
never-push-main rule is unaffected.

- "A unit test reproduces the failure" â†’ implement stage's log confirms the regression test raised `CalledProcessError` before the fix and passes after.

## Review notes

The plan considered a stash-and-restore alternative and rejected it: popping
a stash of `main`'s `prd.json` diff onto a pre-existing work branch with a
divergently committed `prd.json` risks a mid-checkout conflict, which the
bookkeeping-commit approach avoids entirely.
