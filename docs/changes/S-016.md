# S-016 — Phone-facing lifecycle board via GitHub stage labels

## What shipped

GH-sourced tickets now carry a mutually-exclusive `stage:<x>` label
(`stage:plan`, `stage:implement`, `stage:test`, `stage:review`,
`stage:document`) on their source issue that reflects the SDLC stage
currently in flight, so a GitHub Projects board can show a
Plan/Do/Check/Review-gate/Blocked view with no new frontend, API, or
backend. The orchestrator remains the sole GitHub authority: a new
`stage_fn` callback, injected exactly like the existing `progress_fn`,
fires at each stage's *entry* and drives the label swap in
`workflow_runner`. Plain `S-NNN` stories get no calls at all.

## Surface changes

### CLI / workflows

No new commands. `adw.orchestrator.run_ticket` gains an optional
`stage_fn: StageLabelFn | None = None` parameter (`adw/orchestrator.py`),
called once per stage immediately after `state.stage` is set, including
once before the document-stage retry loop in `_run_document_stage`.
`adw.workflow_runner.run_ticket_work` forwards a `stage_fn` parameter
through to `run_ticket`.

### Schemas & data

`adw/workflow_runner.py` adds `STAGE_LABEL_PREFIX = "stage:"` and
`_make_stage_label_fn(story)`: returns `None` when the story has no
source GitHub issue; otherwise returns a closure that, given a stage
name, computes `stage:<x>`, calls the existing `_set_run_label` to add
the new label and remove the previously-set one (if different), and
remembers the new label as `prev` for the next call. This reuses
`_set_run_label`'s best-effort, error-swallowing, S-NNN-no-op behavior
rather than duplicating it.

Both call sites build their own closure: `run_one_story` (sequential
path) and the parallel backlog's per-story `work()` closure each call
`_make_stage_label_fn(story)` independently, so concurrent workers never
share the `prev` state.

## Behavior & breaking changes

None. The new parameter defaults to `None` everywhere, so every existing
caller of `run_ticket` / `run_ticket_work` is unaffected. The stage
label is additive alongside the existing terminal `in-progress` /
`done` / `blocked` / `quotad` run labels and the S-014 per-stage
comments — neither is removed or duplicated.

## How it was verified

- "mutually-exclusive stage label, prior swapped" → `tests/test_workflow_runner.py::test_stage_label_fn_swaps_prior_label` — asserts `stage:plan` added with no removal, then `stage:implement` added with `stage:plan` removed.
- "driven from orchestrator via injected callback, never called by stage agents" → `tests/test_orchestrator.py::test_stage_fn_called_at_entry_in_order` — asserts call order `["plan", "implement", "test", "review", "document"]`; label logic lives only in `workflow_runner`.
- "best-effort, GitHub/credential/offline failure swallowed" → `tests/test_workflow_runner.py::test_stage_label_fn_swallows_github_error` — `get_token` raises `GitHubError`, call does not raise.
- "plain S-NNN no-op; existing labels/comments preserved" → `tests/test_workflow_runner.py::test_stage_label_fn_is_none_for_plain_story` asserts `_make_stage_label_fn` returns `None` for an `S-` story; existing `_set_run_label`/`progress_fn` tests are untouched.
- "README documents human-only Projects setup" → README.md "Lifecycle board (GitHub Projects setup, S-016)" section, label-to-column table, explicit "one-time, out-of-agent-scope human action ... a headless run never attempts it."
- "full suite green" → test stage reported `uv run pytest -q` → 277 passed.

## Review notes

The stage label is intentionally never cleared at terminal outcome
(e.g. `stage:document` and `done` can both be present once a ticket
finishes) — no AC requires terminal cleanup, and the Blocked column is
keyed on the existing `blocked` run label, not a stage label. Firing
`stage_fn` at stage *entry* (rather than the exit-time `progress_fn`) is
deliberate: it's the only way "currently in stage X" is correct instead
of lagging one stage behind.

This branch is stacked on the unmerged S-015 work (parallelization +
usage-limit/quotad handling), so `git diff main...HEAD` also shows
`quotad` outcome handling, worktree support, and budget changes that
are not part of this ticket — this doc covers only the S-016 stage-label
footprint.
