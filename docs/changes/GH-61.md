# GH-61: Plan emits a file manifest so downstream stages stop re-surveying the repo

## What shipped

Each ADW stage runs as a separate cold `claude -p` process with no shared
session, so implement/test/review each re-explored the codebase from scratch —
measured as the dominant per-ticket token cost. This ticket adds a
machine-readable file manifest to the plan stage's status block and inlines it
into the downstream prompts, telling those stages exactly which files to open
instead of surveying the repo themselves.

## Schemas & data

`StatusBlock` (`adw/status.py`) gains an optional `file_manifest: dict | None`
field, parsed leniently in `parse_status_block`: a `{"edit": [...], "read":
[...]}` object round-trips as-is; anything non-dict (or absent) coerces to
`None`. This mirrors the existing `pr_description` field's parse pattern — one
parse path, no schema rejection of unknown/missing keys.

## CLI / workflows

`compose_stage_prompt` (`adw/workflow_runner.py`) gains a `_plan_manifest(run_dir)`
helper: it globs `iter*_plan_output.md`, takes the latest, parses its status
block, and renders an "open only these, do not survey" markdown section from
the manifest's `edit`/`read` lists. The section is appended only when composing
the `implement`, `test`, or `review` prompts (not `plan`, which has no prior
manifest to read).

Degrades to `None` — no section, no error — when: there is no plan output yet,
the plan output has no parseable status block, or the manifest is missing/empty.
A bad or missing manifest costs a stage a little re-exploration; it never
blocks a run.

`commands/PLAN.md` and `stage_specs/plan_feat.md` add the `file_manifest`
template field and document its shape (`edit`/`read` arrays of `path` or
`path:line` strings). `commands/IMPLEMENT.md`, `commands/TEST.md`, and
`commands/REVIEW.md` each add the corresponding rule: open only the manifest's
files, and read further only if the manifest is wrong or insufficient.

## How it was verified

- "Plan status block carries a structured file manifest" → `tests/test_status.py`:
  a `file_manifest` object round-trips into `StatusBlock.file_manifest`; absent
  defaults to `None`; non-dict coerces to `None`.
- "Implement/test/review prompts inline that manifest" → `tests/test_prompts.py::test_compose_stage_prompt_inlines_file_manifest`:
  a plan output carrying a manifest produces an implement prompt containing the
  "open only these...do not survey" section and the listed paths; a plan output
  with no status block composes with no manifest section.
- "Representative ticket shows reduced downstream reads" → mechanism-only:
  the manifest + survey-suppression instruction is delivered and unit-tested,
  but the actual token delta is a live-run observation (non-deterministic
  `claude -p` token counts), not something a unit test can fabricate. This
  run's own plan predated the `PLAN.md` template change, so it emitted no
  manifest itself — the payoff lands on subsequent runs.
- "Full suite stays green" → `uv run pytest -q`: 364 passed.
