# S-006 ‚Äî GitHub Issues intake + done/blocked notifications

## What shipped

Bridges the harness to a phone-facing backlog: `workflows/sync_issues.py`
pulls open GitHub Issues labeled `adw` into `prd.json` stories (intake),
and `adw/workflow_runner.run_one_story` now pushes the work branch,
opens/updates a PR, and comments the outcome on the source issue when a
ticket finishes (outbound). Both directions go through a new
`adw/github.py` stdlib-`urllib` REST client authenticated with the locally
cached git credential ‚Äî no new dependencies, no new services.

## Surface changes

### CLI / workflows

- `workflows/sync_issues.py` ‚Äî `uv run python workflows/sync_issues.py`.
  Fetches open issues labeled `adw`, maps each to a `Story` (id `GH-<n>`,
  type from a `feat`/`bug`/`chore`/`system-repair` label, priority from a
  `p<N>`/`priority-N` label, description/acceptance criteria parsed out of
  the issue body's `## Acceptance Criteria` section), appends new stories
  to `prd.json`, and prints how many were added vs. skipped with reasons.
  Already-synced issues (`GH-<n>` id present) and malformed issues
  (missing/ambiguous type label, no acceptance criteria) are skipped and
  never written.

### Schemas & data

- No `Story`/`Prd` schema change ‚Äî `issue_to_story` produces ordinary
  `Story` records using the existing dataclass.

## Behavior & breaking changes

- `run_one_story` (`adw/workflow_runner.py`) now ends every run ‚Äî `done`
  or not ‚Äî with a call to `_notify_github(story, outcome.outcome,
  outcome.reason or "")`, after the existing outcome-bookkeeping commit.
  This pushes `adw/<story.id>` to `origin` and opens-or-updates a PR
  (head `adw/<id>` ‚Üí base `main`), commenting on the source issue only
  when `source_issue_number(story.id)` resolves (i.e. the story came from
  a `GH-<n>` issue). All of this is best-effort: `_push_branch` catches
  push failures and `_notify_github` catches `GitHubError` from the token
  lookup or any API call, prints a one-line skip message, and returns ‚Äî
  it can never turn a `done` outcome into a failure.

## How it was verified

- "sync_issues.py converts open GitHub Issues ‚Ä¶ deterministic id mapping
  ‚Ä¶ never duplicates" ‚Üí `tests/test_sync_issues.py::test_sync_issues_adds_new`,
  `::test_sync_issues_dedup`, `::test_issue_story_id`.
- "Malformed issues are skipped with a printed reason, never written" ‚Üí
  `tests/test_sync_issues.py::test_sync_issues_skips_malformed` (asserts
  zero stories written) plus the `skip_reason` fixture tests.
- "On ticket completion the workflow opens (or updates) a PR for adw/<id>
  and comments done/blocked + reason on the source issue" ‚Üí
  `tests/test_workflow_runner.py::test_notify_github_with_issue_id_opens_pr_and_comments`
  (asserts `head="adw/GH-42"`, `base="main"`, comment posted),
  `::test_notify_github_without_issue_id_skips_comment` (no comment for
  non-`GH-` ids), and `tests/test_github.py::test_open_or_update_pr_patches_on_422`
  for the update-existing-PR path.
- "All GitHub calls go through the REST API with the locally stored
  credential; degrade with a clear error when offline" ‚Üí
  `tests/test_github.py::test_api_request_offline_raises_github_error`
  (urllib `URLError` ‚Üí `GitHubError`) and
  `tests/test_workflow_runner.py::test_notify_github_swallows_github_error`
  / `::test_notify_github_still_tries_pr_after_push_failure` (notification
  degrades without raising).
- "Unit tests cover the issue-to-story mapping and skip logic with fixture
  payloads (no live API in tests)" ‚Üí `tests/test_sync_issues.py` is fully
  fixture-driven; no network calls.
- "Full suite stays green" ‚Üí `uv run pytest -q` reported 172 passed at
  both the implement and test stages.

## Review notes

- Notification wiring lives once in `run_one_story` rather than in each
  `workflows/*.py` entry point, so it covers feat/bug/trivial/backlog
  workflows uniformly without tripling the call site.
- The pushed ref is always the explicit `adw/<story.id>` branch ‚Äî never a
  bare `git push` ‚Äî so the existing guard against pushing/force-pushing
  `main` is unaffected.
- `tests/test_workflow_runner.py`'s module docstring still describes only
  the S-010 test-evidence verifier even though it now also covers
  `_notify_github`; a future pass could update it, but it doesn't affect
  correctness.
