# GH-51: PR body should follow the repo PR template, authored by the review stage

## What shipped

Harness-opened PRs no longer always restate the ticket description.
The review stage may now emit an optional `pr_description` field in its
status block (only when `exit_signal: true`) ‚Äî a human-facing PR
summary written by the stage that actually read the diff. That value
flows through the existing status-block seam to become the PR body,
falling back to today's ticket-restatement body when absent.

## Behavior & breaking changes

None for existing callers/pipelines. `pr_body()` gained a third,
defaulted parameter, so every current call site keeps producing the
same body as before. Pipelines that skip review (bug skips review;
chore skips plan+review) and any non-done/blocked outcome continue to
get the current ticket-restatement body, since `pr_description` only
gets populated on the gate-success path.

## How it was verified

- `stage_specs/review_feat.md` instructs review to emit `pr_description`
  ‚Üí verified by reading the new "PR description" section (lines 48-56).
- `StatusBlock`/parser and `TicketOutcome` carry it ‚Üí
  `tests/test_status.py::test_parses_pr_description_when_present`,
  `test_pr_description_defaults_to_none_when_absent`;
  `tests/test_orchestrator.py::test_pr_description_carried_from_review_gate`,
  `test_pr_description_none_when_review_omits_it`.
- `pr_body()` renders it when present, keeps the current body otherwise ‚Üí
  `tests/test_github.py::test_pr_body_without_pr_description_is_unchanged`,
  `test_pr_body_renders_pr_description_when_present`.
- Skip-review/non-done pipelines unaffected ‚Üí `pr_description` defaults
  to `None` everywhere except the single gate-success call site in
  `adw/orchestrator.py`.
- Full suite green: `uv run pytest -q` ‚Üí 318 passed.

## Review notes

- Enhancement with a fallback, not a replacement ‚Äî accepted per the
  ticket; no special-casing was added for review-skipping pipelines.
- `pr_description` is captured from the review result before the
  document stage runs, so there's no risk of it being overwritten by a
  later stage's status block.
- The PR body content has no schema enforcement beyond the guidance in
  `stage_specs/review_feat.md`; quality of the summary depends on the
  reviewing agent.

## File map

- `adw/github.py` ‚Äî `pr_body()` takes an optional `pr_description` and renders it when present, else the original body.
- `adw/orchestrator.py` ‚Äî `TicketOutcome.pr_description` field; populated from the review result at the gate-success `_finish` call.
- `adw/status.py` ‚Äî `StatusBlock.pr_description` field, parsed from the status-block JSON.
- `adw/workflow_runner.py` ‚Äî `_notify_github` and `_finalize_story` thread `pr_description` through to `pr_body()`.
- `stage_specs/review_feat.md` ‚Äî documents the new `pr_description` field for the review stage.
- `tests/test_github.py`, `tests/test_orchestrator.py`, `tests/test_status.py` ‚Äî cover the new field end-to-end.
