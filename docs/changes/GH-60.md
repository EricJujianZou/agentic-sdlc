# GH-60: fix usage-limit false positive on `blocked` outcomes

## What shipped

`CircuitBreaker._evaluate` (`adw/safety.py`) gated usage-limit detection on
`not stage_succeeded`, so any non-`success` outcome — including a correct
`blocked` verdict — had its `raw_output`/`stderr` scanned for quota phrases.
A `blocked` plan stage whose prose legitimately mentioned "session limit"
(GH-56, 2026-06-22) tripped the matcher and stamped a bogus 5-hour cooldown,
halting the backlog. The fix narrows the gate to `result.status is None`:
the only state a genuine provider cut-off can produce, since a real quota
halt prevents the stage from returning any parseable structured status block
at all. Any stage that returns a verdict — `success`, `blocked`, `failure`,
etc. — is now trusted not to be a quota halt regardless of its prose.

## Behavior & breaking changes

- `CircuitBreaker._evaluate` no longer treats `blocked` (or other
  non-`success`, non-`None`-status) outcomes as detection candidates.
  Detection now fires only when `result.status is None`. This is strictly
  narrower than before, so no caller that previously got a real quota halt
  loses it — only the false-positive case (`blocked`/non-success with a
  parsed status) stops triggering.

## How it was verified

- "A stage that returns ANY structured outcome must not trip usage-limit
  detection on prose alone" → confirmed by reading the diff: the guard is now
  `result.status is None`, so any parsed status block (any outcome) skips
  detection.
- "A genuine cut-off (`result.status is None`) carrying a usage-limit signal
  still halts" → `uv run pytest -q tests/test_safety.py -k "usage_limit or
  session_limit"`; `test_usage_limit_halts_immediately` and
  `test_session_limit_routes_to_quotad` (both `status=None`) still assert
  `USAGE_LIMIT_HALT_REASON` fires.
- "Unit test: a `blocked` StageResult whose `raw_output` contains 'session
  limit' must not halt" → new test
  `test_blocked_outcome_with_usage_limit_text_does_not_halt`
  (`tests/test_safety.py:305`) asserts `reason is None` and
  `cooldown_until is None`.
- "Full suite stays green" → `uv run pytest -q`: 342 passed.

## Review notes

- This is a recurrence of the 2026-06-19 false positive in a new shape (a
  `blocked` outcome instead of a `success` one); the fix replaces the
  `not stage_succeeded` heuristic with the stricter, semantically-accurate
  `result.status is None` check rather than special-casing `blocked`.
- Filed human-gated (`system-repair`, no `adw` label) on purpose: the fix
  sits in `adw/safety.py`, which the harness itself is not permitted to
  edit, so it could not safely dogfood this fix.
- Companion finding (not part of this ticket): GH-56 is mis-typed `feat`
  when its blocked reason was a harness-file edit restriction — left as-is
  per ticket scope.
