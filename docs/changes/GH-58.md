# GH-58: New ticket branches were cut from HEAD instead of main

## What shipped

`_ensure_work_branch` (`adw/workflow_runner.py`) previously created a *new*
ticket branch from whatever HEAD currently pointed at. Since the harness
leaves HEAD on the last ticket's branch between runs — and an interrupted run
leaves HEAD on the branch of the ticket that crashed — the next ticket's
branch was silently stacked on top of a stranded/dead branch, poisoning its
diff-to-`main`. The fix changes only the new-branch (`else`) arm to cut from a
clean base (`origin/main` after a best-effort fetch, else local `main`)
instead of HEAD. The resume arm (checkout of an existing `adw/<id>` branch) is
untouched.

## Behavior & breaking changes

- New ticket branches are now created with `git checkout -f -b <branch> <base>`
  instead of `git checkout -b <branch>`, where `<base>` is `origin/main` (if an
  `origin` remote exists and the ref resolves after a best-effort fetch) or
  local `main` as fallback. If neither ref exists, behavior falls back to the
  old HEAD-relative checkout (degraded mode, no main available).
- A plain `checkout -b <base>` would abort whenever HEAD is on a divergent
  branch with an uncommitted `prd.json` (the common intake path, since
  `sync_issues.py` rewrites `prd.json` uncommitted before branch creation).
  To avoid regressing that path, any tracked-dirty files are snapshotted
  (`_snapshot_tracked_dirty`) before the forced checkout and restored
  byte-for-byte after, so the sync write lands as an uncommitted change on
  the new branch for `_mark_in_progress` to commit. Tracked file *deletions*
  are not preserved by the snapshot (only modifications/additions) — acceptable
  since sync only ever rewrites `prd.json`.

## How it was verified

- "A new ticket branch is created from `main`/`origin/main`, independent of
  current HEAD" → `test_ensure_work_branch_new_branch_cut_from_main_not_head`:
  asserts the new branch's tip equals `main`'s tip after `_ensure_work_branch`
  is called while HEAD sits on an unrelated `adw/dead` branch.
- "Resuming an existing `adw/<id>` branch is unchanged" →
  `test_ensure_work_branch_commits_dirty_sync_write_before_resuming` (existing
  GH-46 test) still passes unmodified — the resume arm was not touched.
- "An interrupted run that leaves HEAD on a dead ticket branch cannot poison
  the next" → both new tests assert
  `git merge-base --is-ancestor adw/dead adw/GH-58` returns non-zero.
- "Unit test covers: new branch cut while HEAD is on an unrelated branch ends
  up based on main" →
  `test_ensure_work_branch_new_branch_cut_from_main_not_head` and
  `test_ensure_work_branch_new_branch_preserves_dirty_prd_from_dead_head`.
- "Full suite stays green" → `uv run pytest -q` reported 343 passed (test
  stage); review stage could not independently re-run pytest (permission-
  blocked that session) and relied on this evidence plus a direct code read.

## Review notes

- `_snapshot_tracked_dirty` calls `subprocess.run` directly rather than the
  module's `_git()` helper, because `_git()` strips the whole output, which
  would eat the leading status byte of the first `git status --porcelain`
  line and shift the fixed-column parse.
- The branch this work shipped on (`adw/GH-58`) was itself cut from a
  poisoned HEAD before this fix existed — its diff-to-`main` carries
  unrelated commits (a CLAUDE.md `\ADW\` power-settings note, an
  S-017→GH-59 `prd.json` reshuffle, and normal in_progress/sync bookkeeping)
  on top of the substantive change. The merger should squash with that in
  mind; the GH-58-specific change is confined to `adw/workflow_runner.py` and
  `tests/test_workflow_runner.py`.
