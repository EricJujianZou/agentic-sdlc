# GH-79 — Shard prd.json into one file per ticket

## Problem

`prd.json` was a single file holding the state of **every** ticket (each story's
`status`/`passes`). As the harness works a ticket it commits state changes to
that file on the ticket's branch ("mark GH-61 in_progress", "record GH-61 done").
Run two tickets and both branches edit the **same** `prd.json`; merging both PRs
into `main` produces a git merge conflict a human must resolve by hand every
time. It is structural — all ticket state funnels through one shared file.

## Decision (sharding)

Split the one `prd.json` into **one file per ticket** under a `prd/` directory:

```
prd/
  _meta.json      {"project": "agentic-sdlc"}   # single global label
  GH-61.json      the full Story (all 9 fields)
  S-001.json
```

Two branches that edit different stories then touch different files, so git
merges every ticket PR cleanly. A flat one-file-per-story layout was chosen over
a per-issue *folder* (`prd/GH-61/story.json`) as YAGNI — nothing concrete needs a
second per-ticket file, and flat kills the conflict equally.

`project` is preserved but is a **non-functional label**: the target repo is
resolved by `adw/paths.py`/`ADW_REPO`, not `prd.project`. It is written once at
migration and never mutated per-ticket, so the shared `_meta.json` is never a
conflict source.

## Change

- `adw/tickets.py`
  - `load_prd(path)` / `save_prd(prd, path)` keep their signature (`path` is the
    canonical `prd.json` anchor) but now read/write shards under the sibling
    `prd/` directory. The in-memory `Prd`/`Story` shape is unchanged, so the
    ~120 call sites that use `prd.stories`/`mark_story` are untouched.
  - Each story serializes to `prd/<id>.json` deterministically (stable dataclass
    field order, `ensure_ascii=True`) via the existing atomic-write helper, so an
    unchanged story writes byte-identical content → no git diff → no commit churn
    → nothing to conflict on. `save_prd` removes shards for dropped stories.
  - `load_prd` reconstructs `stories` sorted by `id` (order is not stored on
    disk; the pickers re-sort by `(priority, id)` themselves). The `id` lives in
    the file (source of truth); the filename is asserted to match as an integrity
    check on hand-edits.
  - **Legacy fallback:** if the `prd/` directory is absent, `load_prd` reads a
    pre-sharding single-file `prd.json` (read-only), so unmigrated branches and
    target repos still load until they drain.
- `hooks/pretooluse_guard.py`: `is_system_repair_run` reads the active ticket's
  type from its `prd/<id>.json` shard (falling back to legacy `prd.json`).
- `dashboard/index.html`: loads stories from the `prd/` directory listing served
  by the static file server, falling back to `../prd.json` for an unmigrated repo.
- Migrated this repo's `prd.json` → `prd/` (30 shards + `_meta.json`). The legacy
  `prd.json` is kept as a read-only **tombstone** so the ~15 unmerged `adw/GH-*`
  branches still load; a later cleanup PR removes the fallback + tombstone once
  they drain. Other repos the engine manages (design-system, scholarship-factory)
  auto-migrate on their next harness run via the new `save_prd`.

## Tests

`uv run pytest -q` → 389 passed. New coverage in `test_tickets.py`: sharded
on-disk layout, shards-win-over-stale-tombstone, legacy fallback, missing-store
raises, orphan-shard removal, id/filename-mismatch integrity, and the acceptance
property — **two ticket branches that each touch only their own story merge with
no conflict** (`test_independent_story_branches_merge_without_conflict`). Updated
the two layout-asserting tests in `test_tickets.py`/`test_paths.py` and the two
`reap_stale_in_progress` tests (read saved state via `load_prd`, not the raw
tombstone file).

## Limitation / follow-up

The legacy `prd.json` tombstone and the read-only fallback in `load_prd`/the
dashboard are transitional. Once the unmerged `adw/GH-*` branches drain, a
cleanup PR should delete `prd.json` and the fallback paths. Until then, an old
branch that merges may produce a one-time cosmetic conflict on the frozen
tombstone — the live state lives in `prd/`, so it is harmless.
