# GH-78 ‚Äî Scheduler terminal-guard: never re-run already-merged tickets

## Problem

The scheduler re-selected and re-ran tickets whose work was already completed
and merged to `main`, wasting a full plan‚Üíimplement cycle and opening spurious
PRs. Two confirmed instances: **GH-61** (merged via PR #68, re-run to `blocked`,
spurious PR #77) and **GH-56** (merged via PR #67, re-run to `done`, spurious
PR #76). GH-56 began life as a `feat`, so the bug is **ticket-type-agnostic**.

## Root cause

Two factors let an already-merged ticket be re-selected:

1. **No terminal-guard.** `pick_next_story` trusts the local `prd.json`. When
   that ledger drifts behind `origin/main` (a `reclaim`/reopen resets a merged
   ticket to a pickable state), nothing checks that the work is already merged.
2. **`reclaim_stale_in_progress` bypassed the human-gate.** It flipped *every*
   stale `in_progress` story to `open`, including `system-repair` tickets that
   are supposed to stay human-gated at `blocked`.

## Change

- `adw/tickets.py`
  - `reclaim_stale_in_progress` now restores a stale `system-repair` ticket to
    `blocked` (human-gated), not `open`. Every other type still returns to `open`.
  - New pure `reconcile_completed(prd, completed_ids)`: forces any story whose
    id is in `completed_ids` to terminal (`passes=true`, `status="done"`).
- `adw/workflow_runner.py`
  - New `_completed_ticket_ids_on_main()`: best-effort read of
    `docs/changes/<id>.md` on `origin/main` (the merged-completion artifact);
    format-independent, so it survives the GH-79 prd reshape.
  - New `reconcile_completed_against_main(...)`: loads the prd, reconciles
    against the merged set, persists + commits. `completed_ids` is injectable
    for tests.
- `workflows/poll_all.py`, `workflows/run_backlog.py`: call
  `reconcile_completed_against_main()` right after the stale-reclaim step and
  before selection, so a stale ledger can never re-pick merged work.

## Tests

`uv run pytest -q` ‚Üí 385 passed. New coverage: `system-repair` reclaim ‚Üí
`blocked`; `reconcile_completed` (marks merged terminal, idempotent, ignores
unknown ids, only touches matches); the wrapper's persist + empty-set
short-circuit.

## Limitation / follow-up

The merged signal is the `docs/changes/<id>.md` artifact; a completed ticket
that never produced one would not be caught by reconcile (the reclaim-gate fix
still applies). Re-doing already-merged work should use a **new** ticket id per
the naming convention, not a re-run of the merged id.
