# GH-47: Interrupted run strands a ticket in `in_progress` with no auto-recovery

## What shipped

An interrupted backlog pass (machine sleep, killed agent, a concurrent tree
operation) previously left a ticket at `status="in_progress"` forever — `pick_next_story`/`pick_next_stories` only return `open`/`quotad` candidates, so a
stranded ticket was silently orphaned and required manual cleanup. This ticket
adds a **reaper** that runs once at the start of each backlog pass: it flips
any `in_progress` story back to `open` unless it can prove a run is actively
working that exact ticket (a fresh `state.json` heartbeat naming it). The
threshold is configurable via `configs/budgets.json`.

## CLI / workflows

`workflows/run_backlog.py:53-58` — `run_backlog` now calls
`reap_stale_in_progress(stale_seconds=budgets.get("in_progress_stale_minutes", 60) * 60)`
once, before the `if parallel:` branch (so it covers both sequential and
parallel dispatch), inside the poll's existing single-flight lock. If any
ticket was reclaimed it prints `backlog runner: reclaimed stale in_progress
ticket(s): <ids>`.

## Schemas & data

`configs/budgets.json` — added `"in_progress_stale_minutes": 60` (4x the
existing 15-minute `stage_timeout_minutes`), the configurable staleness
threshold.

## Behavior & breaking changes

- `adw/tickets.py` — new pure predicate `reclaim_stale_in_progress(prd, *,
  stale_seconds, live_ticket_id, state_age_seconds) -> list[str]`. For every
  `in_progress` story, it is left alone only if `live_ticket_id == story.id`
  **and** `state_age_seconds <= stale_seconds`; otherwise it is flipped to
  `open` and its id is returned. A missing/corrupt heartbeat (`None` args) is
  never treated as live, so the safe default is to reclaim.
- `adw/workflow_runner.py` — new IO wrapper `reap_stale_in_progress(*,
  stale_seconds, prd_path=None, state_path=None) -> list[str]`. Reads
  `state.json`'s `ticket_id` and mtime age (swallowing
  `OSError`/`ValueError`/`KeyError`/`json.JSONDecodeError` as "no live
  ticket"), loads `prd.json`, calls the predicate, and — only if something was
  reclaimed — saves the prd and commits via `_commit_bookkeeping`. A clean
  tree with nothing stale produces no commit.
- `adw/workflow_runner.py:_ensure_work_branch` — also picked up a small
  related fix from the stacked GH-46 branch beneath this one: it now commits
  any pending sync write before `git checkout` onto an existing branch, so the
  checkout doesn't abort on a dirty tree.

## Review notes

- This branch is stacked on `adw/GH-46` (status `done`, unmerged), so the
  diff to `main` also bundles GH-46's `_ensure_work_branch` fix and its own
  change doc — that's the repo's standard stacked-branch pattern, not part of
  this ticket's scope.
- Known, accepted limitation (flagged in the plan's risk note, not a
  must-fix): the heartbeat is the single per-repo `state.json`, so under
  Tier-1 parallel worktrees a live worktree-local run isn't visible to the
  reaper running against the main tree's `state.json`. The 60-minute
  threshold keeps the false-reclaim window wide for the sequential poll path
  this ticket targets; a worktree-aware heartbeat is a candidate follow-up if
  parallel mode reclaims a ticket while it's still running.
- The review stage could not independently re-run `uv run pytest -q` in its
  session (permission-gated) and relied on the test stage's evidence instead.
